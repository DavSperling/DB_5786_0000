## 2. Reverse-engineering algorithm (DSD -> ERD)

> *The algorithm below is the one applied to the received backup (`backup2.sql`) to reconstruct group 2's ERD.*

```
ALGORITHM : Reverse_Engineer_DSD_to_ERD
INPUT     : a PostgreSQL backup file (DDL + data)
OUTPUT    : an ERD diagram (entities, relationships, cardinalities)

STEP 1 — Extract tables (DSD)
   For each CREATE TABLE T:
       parse the columns, types, CHECK, NOT NULL, UNIQUE
       parse the PRIMARY KEY
       parse the FOREIGN KEYs (declared inline or via ALTER TABLE ADD CONSTRAINT)

STEP 2 — Identify entities
   Each table T that is NOT a pure junction table
   (i.e. whose PK is NOT exclusively composed of 2 FKs)
   becomes an entity in the ERD.

STEP 3 — Identify binary relationships
   For every FK in T(c) -> T'(c'):
       create a relationship R between entity T and entity T'
       cardinality on T' side:
           - "1" always (the FK references a PK)
       cardinality on T side:
           - "1" if column c is UNIQUE      (1-to-1)
           - "N" otherwise                  (N-to-1)
       participation:
           - mandatory (solid line) if NOT NULL
           - optional  (dashed line) if NULLABLE

STEP 4 — Identify N-N relationships
   If T is a junction table (PK = (FK1, FK2)):
       replace T with an N-to-N relationship between the 2 referenced tables.
       remaining columns of T become attributes of the relationship.

STEP 5 — Identify weak entities
   An entity E is weak if:
       - its PK contains a FK to another entity E'
       - its PK alone is not enough to identify it
   Represent E with a double-bordered rectangle and the identifying
   relationship with a double diamond.

STEP 6 — Restore attributes and their properties
   On every entity, draw the simple attributes.
   Underline the attributes that compose the PK.

STEP 7 — Verify consistency
   Compare the diagram against the backup data
   (e.g. SELECT COUNT(*) on every table, make sure no orphan
   relationship exists).
```

### Application to group 2's backup — result

| Entity                | PK                | FK                                | Type     |
|-----------------------|-------------------|-----------------------------------|----------|
| `customer`            | `customer_id`     | —                                 | Strong   |
| `loyalty_tier`        | `tier_id`         | —                                 | Strong (lookup) |
| `reason`              | `reason_id`       | —                                 | Strong (lookup) |
| `status_type`         | `status_id`       | —                                 | Strong (lookup) |
| `loyalty`             | `loyalty_id`      | `customer_id` (UNIQUE), `tier_id` | Strong; 1-1 with `customer` |

## 3. Integration decisions (integrated ERD)

### 3.1 Chosen integration pivot
**`"ORDER".customer_id -> customer.customer_id`**

This is the only natural bridge between the two schemas: our `"ORDER"` table already had a `customer_id` column (Mockaroo-generated in Stage 1, but without an FK), and the imported department brings the `customer` table that defines the Customer entity.

### 3.2 Documented decisions

| #  | Decision | Rationale |
|----|----------|-----------|
| **D1** | No table renamed, no merge. | The 6 local tables and the 9 imported tables are all kept as-is, complying with the requirement "do not recreate tables". No name conflict: our tables (`"ORDER"`, `BILL`, `PAYMENT`, ...) and theirs (`customer`, `reservation`, ...) have disjoint names. |
| **D2** | Re-mapping of invalid `customer_id` values in `"ORDER"` via `((customer_id - 1) % 500) + 1`. | Mockaroo had generated IDs between 1 and 999 974, only a few of which existed among the 500 actual customers. The modulo uniformly distributes orders across the 500 existing customers, with no row loss and in a deterministic (replayable) way. |
| **D3** | Add a **single** foreign key: `"ORDER".customer_id -> customer.customer_id`. | This is what materialises the integration at the ERD level. No other cross FK is introduced (no `reservation_id` on `"ORDER"`, no replacement of `order_status` with an FK to `status_type`) — option A deliberately stays lightweight. |
| **D4** | `"ORDER".customer_id` remains **NULLABLE**. | Consistent with the original ERD (an anonymous/walk-in order is allowed). An FK accepts NULLs. |
| **D5** | Transfer technique: `postgres_fdw`. | More instructive than `pg_dump`/`psql -f`, and it is the technique used by the reference project (`eliezergensburger/DBProject_7061_3114`). Foreign tables are **dropped at the end of the script**: we are doing full integration (method A), not method B. |
| **D6** | All CHECK / UNIQUE / FK constraints from the imported schema are reproduced locally. | Guarantees the exact referential integrity of group 2. |
| **D7** | Import order: lookups -> `customer` -> linked tables. | Mandatory for the FKs to validate during the `INSERT INTO ... SELECT` calls. |

## 4. Integration process — command walkthrough

The `Integrate.sql` file is structured in **5 phases**.

### Phase 0 — Idempotent cleanup
```sql
ALTER TABLE "ORDER" DROP CONSTRAINT IF EXISTS fk_order_customer;
DROP TABLE IF EXISTS feedback CASCADE;
-- ... other DROP TABLE IF EXISTS in reverse FK order
DROP SERVER IF EXISTS other_group_server CASCADE;
```
**Purpose**: lets the script be replayed without errors (useful while developing).

### Phase 1 — Foreign Data Wrapper setup
```sql
CREATE EXTENSION IF NOT EXISTS postgres_fdw;
CREATE SERVER other_group_server FOREIGN DATA WRAPPER postgres_fdw
  OPTIONS (host 'localhost', dbname 'other_group_db', port '5432');
CREATE USER MAPPING FOR admin SERVER other_group_server
  OPTIONS (user 'admin', password 'admin123');
```
**Purpose**: lets `restaurant_db` see tables from `other_group_db` as if they were local (via **foreign tables**).

### Phase 2 — Import the 9 tables (repetitive pattern)
For each table T of the imported department:
```sql
CREATE FOREIGN TABLE T_remote (...) SERVER other_group_server
    OPTIONS (schema_name 'public', table_name 'T');

CREATE TABLE T (... columns + constraints ...);

INSERT INTO T SELECT * FROM T_remote;
```
**Purpose**: the `foreign table` is a virtual mirror; we read its rows through an `INSERT INTO ... SELECT` that physically copies them into the real local table.

**Exact order**: `loyalty_tier` -> `reason` -> `status_type` -> `customer` -> `loyalty` -> `loyalty_transaction` -> `reservation` -> `feedback` -> `waitlist` (respects the FK dependencies).

### Phase 3 — Integration link
```sql
UPDATE "ORDER"
SET    customer_id = ((customer_id - 1) % 500) + 1
WHERE  customer_id IS NOT NULL;

ALTER TABLE "ORDER"
ADD CONSTRAINT fk_order_customer
FOREIGN KEY (customer_id) REFERENCES customer(customer_id);
```
**Purpose**: (1) make every `customer_id` valid; (2) declare the FK that materialises the integration at the schema level.

### Phase 4 — Cleanup
```sql
DROP FOREIGN TABLE customer_remote;
DROP FOREIGN TABLE loyalty_tier_remote;
-- ... 9 DROP statements in total
```
**Purpose**: the bridge is no longer needed, and it is best not to leave live links to another database (security + cleanliness).

### Phase 5 — Checks
A final query lists `COUNT(*)` for every table to make sure the integration is complete.

### VIEW 1 — `Customer_Order_Summary`  *(Orders & Billing perspective)*

**Description**:
For each customer, aggregates their commercial activity: number of orders, total billed, average ticket, last order date. Joins **3 tables**: `customer` + `"ORDER"` + `bill`.

**Definition**:
```sql
CREATE VIEW Customer_Order_Summary AS
SELECT
    c.customer_id,
    c.first_name || ' ' || c.last_name AS customer_name,
    c.email,
    COUNT(DISTINCT o.order_id)                 AS total_orders,
    COALESCE(SUM(b.final_amount), 0)           AS total_spent,
    COALESCE(ROUND(AVG(b.final_amount), 2), 0) AS avg_bill_amount,
    MAX(o.order_time)                          AS last_order_time

LEFT JOIN  "ORDER" o ON c.customer_id = o.customer_id
```

### VIEW 2 — `Customer_Loyalty_Status`  *(Customer / Loyalty perspective)*

**Description**:
Exposes for every customer their loyalty tier, points balance, number of point movements, and number of past reservations. Joins **4 tables**: `customer` + `loyalty` + `loyalty_tier` + `reservation` (`loyalty_transaction` is used only for the COUNT).

**Definition**:
```sql
CREATE VIEW Customer_Loyalty_Status AS
SELECT
    c.customer_id,
    c.first_name || ' ' || c.last_name           AS customer_name,
    COALESCE(lt.level, 'No Loyalty')             AS tier_level,
    COALESCE(l.points, 0)                        AS current_points,
    COUNT(DISTINCT lx.transaction_id)            AS loyalty_tx_count,
    COUNT(DISTINCT r.reservation_id)             AS total_reservations,
    MAX(r.datetime)                              AS last_reservation_date
FROM        customer c
LEFT JOIN   loyalty             l  ON c.customer_id = l.customer_id
LEFT JOIN   loyalty_tier        lt ON l.tier_id     = lt.tier_id
LEFT JOIN   loyalty_transaction lx ON l.loyalty_id  = lx.loyalty_id
LEFT JOIN   reservation         r  ON c.customer_id = r.customer_id
GROUP BY    c.customer_id, c.first_name, c.last_name, lt.level, l.points;
```

### VIEW 3 — `Customer_Cross_Activity`  *(integrated view)*

**Description**:
Crosses the **two worlds**: number of reservations (Customer/Loyalty side) and number of orders + revenue (Orders/Billing side), enriched with the loyalty tier, plus an `engagement_type` classification. Joins **6 tables** in total: `customer` + `loyalty` + `loyalty_tier` + `reservation` + `"ORDER"` + `bill`.

**Technical note**: we use `LATERAL` subqueries to pre-aggregate orders and reservations independently, which avoids the cartesian product (each bill multiplied by each reservation) that would inflate `total_revenue`.

**Definition**:
```sql
CREATE VIEW Customer_Cross_Activity AS
SELECT
    c.customer_id,
    c.first_name || ' ' || c.last_name              AS customer_name,
    COALESCE(lt.level, 'No Loyalty')                AS tier_level,
    res.nb_reservations,
    ord.nb_orders,
    ord.total_revenue,
    CASE
        WHEN ord.nb_orders   = 0 AND res.nb_reservations > 0 THEN 'Reservation only'
        WHEN ord.nb_orders   > 0 AND res.nb_reservations = 0 THEN 'Order only'
        WHEN ord.nb_orders   > 0 AND res.nb_reservations > 0 THEN 'Both'
        ELSE 'Inactive'
    END                                             AS engagement_type
FROM        customer c
LEFT JOIN   loyalty      l  ON c.customer_id = l.customer_id
LEFT JOIN   loyalty_tier lt ON l.tier_id     = lt.tier_id
LEFT JOIN LATERAL (
    SELECT COUNT(*) AS nb_reservations
    FROM   reservation r
    WHERE  r.customer_id = c.customer_id
) res ON TRUE
LEFT JOIN LATERAL (
    SELECT COUNT(*)                         AS nb_orders,
           COALESCE(SUM(b.final_amount), 0) AS total_revenue
    FROM   "ORDER" o
    LEFT JOIN bill b ON o.order_id = b.order_id
    WHERE  o.customer_id = c.customer_id
) ord ON TRUE;
```
