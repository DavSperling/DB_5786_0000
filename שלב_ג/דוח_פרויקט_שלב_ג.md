> **Département local (votre dept)** : ORDERS & BILLING (`restaurant_db`)
> **Département importé (groupe 2)** : CUSTOMER / RESERVATIONS / LOYALTY (`other_group_db`)
> **Méthode d'intégration retenue** : א — intégration totale dans une seule base, technique `postgres_fdw` comme pont de transfert.

## 1. Schémas de départ

### 1.1 Mon département (ORDERS & BILLING) — rappel
6 tables :
- `DISCOUNT` (référentiel)
- `"ORDER"` (commande client) — possède un champ `customer_id` non lié initialement
- `ORDER_ITEM` → FK `ORDER`
- `BILL` → FK `ORDER`
- `PAYMENT` → FK `BILL`
- `BILL_DISCOUNT` → FK `BILL`, `DISCOUNT`

### 1.2 Département importé (groupe 2)
9 tables (toutes en minuscules, créées sous PostgreSQL) :
- **Référentiels (sans FK sortantes)** : `loyalty_tier`, `reason`, `status_type`
- **Pivot** : `customer`
- **Liées à `customer`** : `loyalty`, `reservation`, `waitlist`
- **Liées à `loyalty`** : `loyalty_transaction` (FK → `loyalty`, `reason`)
- **Liées à `reservation`** : `feedback` (FK 1-à-1 vers `reservation`)
- **Liées à `status_type`** : `reservation`, `waitlist`

## 2. Algorithme de **reverse engineering** (DSD → ERD)

> *L'algorithme ci-dessous est ce qui a été appliqué au backup reçu (`backup2.sql`) pour reconstruire l'ERD du département groupe 2.*

```
ALGORITHME : Reverse_Engineer_DSD_to_ERD
ENTRÉE      : un fichier de backup PostgreSQL (DDL + données)
SORTIE      : un diagramme ERD (entités, relations, cardinalités)

ÉTAPE 1 — Extraire les tables (DSD)
   Pour chaque CREATE TABLE T :
       lire la liste des colonnes, types, CHECK, NOT NULL, UNIQUE
       lire la PRIMARY KEY
       lire les FOREIGN KEY (déclarées en colonne ou en ALTER TABLE ADD CONSTRAINT)

ÉTAPE 2 — Identifier les entités
   Chaque table T qui n'est PAS une pure table de jonction
   (= dont la PK n'est pas exclusivement composée de 2 FK)
   devient une entité dans l'ERD.

ÉTAPE 3 — Identifier les relations binaires
   Pour chaque FK de T(c) → T'(c') :
       créer une relation R entre l'entité T et l'entité T'
       cardinalité côté T' :
           - "1" toujours (la FK référence une PK)
       cardinalité côté T :
           - "1" si la colonne FK est UNIQUE  (1-à-1)
           - "N" sinon                        (N-à-1)
       caractère obligatoire :
           - obligatoire (ligne pleine) si NOT NULL
           - optionnel  (ligne pointillée) si NULLABLE

ÉTAPE 4 — Identifier les relations N-N
   Si T est une table de jonction (PK = (FK1, FK2)) :
       remplacer T par une relation N-à-N entre les 2 tables référencées.
       les autres colonnes de T deviennent des attributs de la relation.

ÉTAPE 5 — Identifier les entités faibles
   Une entité E est faible si :
       - sa PK contient une FK vers une autre entité E'
       - sa PK ne suffit pas à l'identifier seule
   Représenter E avec un cadre double et la relation identifiante avec un losange double.

ÉTAPE 6 — Restituer les attributs et leurs propriétés
   Sur chaque entité, dessiner ses attributs simples.
   Souligner les attributs qui composent la PK.

ÉTAPE 7 — Vérifier la cohérence
   Confronter le diagramme aux données du backup
   (ex. SELECT COUNT(*) sur chaque table, vérifier qu'il n'existe pas de
   relation orpheline).
```

### Application au backup du groupe 2 — résultat

| Entité           | PK                | FK                                | Type      |
|------------------|-------------------|-----------------------------------|-----------|
| `customer`       | `customer_id`     | —                                 | Forte     |
| `loyalty_tier`   | `tier_id`         | —                                 | Forte (référentiel) |
| `reason`         | `reason_id`       | —                                 | Forte (référentiel) |
| `status_type`    | `status_id`       | —                                 | Forte (référentiel) |

## 3. Décisions d'intégration (ERD intégré)

### 3.1 Pivot d'intégration choisi
**`"ORDER".customer_id → customer.customer_id`**

C'est le seul pont naturel entre les deux schémas : notre table `"ORDER"` possédait déjà un champ `customer_id` (généré par Mockaroo en שלב א, mais sans FK), et le département importé apporte la table `customer` qui définit l'entité Client.

### 3.2 Décisions documentées

| # | Décision | Justification |
|---|----------|---------------|
| **D1** | Aucune table renommée, aucune fusion. | Les 6 tables locales et les 9 tables importées sont toutes conservées telles quelles → respect de la consigne « ne pas recréer les tables ». Aucun conflit de nom : nos tables (`"ORDER"`, `BILL`, `PAYMENT`…) et leurs tables (`customer`, `reservation`…) ont des noms disjoints. |
| **D2** | Re-mapping des `customer_id` invalides dans `"ORDER"` via `((customer_id - 1) % 500) + 1`. | Mockaroo avait généré des IDs entre 1 et 999 974, dont seuls quelques-uns existaient parmi les 500 clients réels. Le modulo distribue uniformément les commandes sur les 500 clients existants, sans perte de ligne et de manière déterministe (re-jouable). |
| **D3** | Ajout d'une **unique** clé étrangère : `"ORDER".customer_id → customer.customer_id`. | C'est ce qui matérialise l'intégration au niveau de l'ERD. Aucune autre FK croisée n'est introduite (pas de `reservation_id` sur `"ORDER"`, pas de remplacement de `order_status` par FK vers `status_type`) — option A volontairement légère. |
| **D4** | `"ORDER".customer_id` reste **NULLABLE**. | Cohérent avec l'ERD initial (un order anonyme/walk-in est possible). Une FK accepte les NULL. |
| **D5** | Méthode technique de transfert : `postgres_fdw`. | Plus pédagogique que `pg_dump`/`psql -f`, et c'est la technique du projet de référence (`eliezergensburger/DBProject_7061_3114`). Les foreign tables sont **droppées en fin de script** : on est bien en intégration totale (méthode א), pas en méthode ב. |
| **D6** | Les contraintes CHECK / UNIQUE / FK du schéma importé sont **toutes** reproduites localement. | Garantit l'intégrité référentielle exacte du groupe 2. |
| **D7** | Ordre d'import : référentiels → `customer` → tables liées. | Indispensable pour que les FK passent au moment des `INSERT INTO ... SELECT`. |

## 4. Processus d'intégration — explication des commandes

Le fichier `Integrate.sql` est structuré en **5 phases**.

### Phase 0 — Nettoyage idempotent
```sql
ALTER TABLE "ORDER" DROP CONSTRAINT IF EXISTS fk_order_customer;
DROP TABLE IF EXISTS feedback CASCADE;
-- ... autres DROP TABLE IF EXISTS dans l'ordre inverse des FK
DROP SERVER IF EXISTS other_group_server CASCADE;
```
**But** : permettre de relancer le script sans erreur (utile pendant le développement).

### Phase 1 — Setup du Foreign Data Wrapper
```sql
CREATE EXTENSION IF NOT EXISTS postgres_fdw;
CREATE SERVER other_group_server FOREIGN DATA WRAPPER postgres_fdw
  OPTIONS (host 'localhost', dbname 'other_group_db', port '5432');
CREATE USER MAPPING FOR admin SERVER other_group_server
  OPTIONS (user 'admin', password 'admin123');
```
**But** : permettre à `restaurant_db` de voir des tables de `other_group_db` comme si elles étaient locales (via des **foreign tables**).

### Phase 2 — Import des 9 tables (motif répétitif)
Pour chaque table T du département importé :
```sql
CREATE FOREIGN TABLE T_remote (...) SERVER other_group_server
    OPTIONS (schema_name 'public', table_name 'T');

CREATE TABLE T (... colonnes + contraintes ...);

INSERT INTO T SELECT * FROM T_remote;
```
**But** : la `foreign table` est un **miroir** virtuel ; on lit ses lignes via un `INSERT INTO ... SELECT` qui les copie physiquement dans la vraie table locale.

**Ordre exact** : `loyalty_tier` → `reason` → `status_type` → `customer` → `loyalty` → `loyalty_transaction` → `reservation` → `feedback` → `waitlist` (respecte les dépendances FK).

### Phase 3 — Lien d'intégration
```sql
UPDATE "ORDER"
SET    customer_id = ((customer_id - 1) % 500) + 1
WHERE  customer_id IS NOT NULL;

ALTER TABLE "ORDER"
ADD CONSTRAINT fk_order_customer
FOREIGN KEY (customer_id) REFERENCES customer(customer_id);
```
**But** : (1) rendre tous les `customer_id` valides ; (2) déclarer la FK qui matérialise l'intégration au niveau du schéma.

### Phase 4 — Cleanup
```sql
DROP FOREIGN TABLE customer_remote;
DROP FOREIGN TABLE loyalty_tier_remote;
-- ... 9 DROP au total
```
**But** : on n'a plus besoin du pont, et il vaut mieux ne pas laisser des liens vivants vers une autre base (sécurité + propreté).

### Phase 5 — Contrôles
Une requête finale liste les `COUNT(*)` de toutes les tables pour vérifier que l'intégration est complète.

### 🟣 VIEW 3 — `Customer_Cross_Activity`  *(vue intégrée)*

**Description** :
Croise les **deux mondes** : nombre de réservations (côté Customer/Loyalty) et nombre de commandes + revenu (côté Orders/Billing), enrichi du tier de fidélité, plus une catégorisation `engagement_type`. Joint au total **6 tables** : `customer` + `loyalty` + `loyalty_tier` + `reservation` + `"ORDER"` + `bill`.

**Note technique** : on utilise des sous-requêtes `LATERAL` pour pré-agréger commandes et réservations indépendamment, ce qui évite le produit cartésien (chaque bill multiplié par chaque réservation) qui fausserait `total_revenue`.

**Définition** :
```sql
CREATE VIEW Customer_Cross_Activity AS
SELECT
    c.customer_id,
    c.first_name || ' ' || c.last_name              AS customer_name,
    COALESCE(lt.level, 'No Loyalty')                AS tier_level,
    res.nb_reservations,
    ord.nb_orders,
    ord.total_revenue,
    CASE
        WHEN ord.nb_orders   = 0 AND res.nb_reservations > 0 THEN 'Reservation only'
        WHEN ord.nb_orders   > 0 AND res.nb_reservations = 0 THEN 'Order only'
        WHEN ord.nb_orders   > 0 AND res.nb_reservations > 0 THEN 'Both'
        ELSE 'Inactive'
    END                                             AS engagement_type
FROM        customer c
LEFT JOIN   loyalty      l  ON c.customer_id = l.customer_id
LEFT JOIN   loyalty_tier lt ON l.tier_id     = lt.tier_id
LEFT JOIN LATERAL (
    SELECT COUNT(*) AS nb_reservations
    FROM   reservation r
    WHERE  r.customer_id = c.customer_id
) res ON TRUE
LEFT JOIN LATERAL (
    SELECT COUNT(*)                         AS nb_orders,
           COALESCE(SUM(b.final_amount), 0) AS total_revenue
    FROM   "ORDER" o
    LEFT JOIN bill b ON o.order_id = b.order_id
    WHERE  o.customer_id = c.customer_id
) ord ON TRUE;
```

**Lecture** : 321 clients (64 %) sont actifs des deux côtés, 179 ont réservé sans jamais commander → cible idéale pour un coupon de bienvenue à utiliser sur place.

#### 🔍 Query 3.2 — Engagement et revenu moyen par tier de fidélité
**But** : vérifier la corrélation entre statut Platinum et revenu réel généré par le client.
```sql
SELECT tier_level,
       COUNT(*)                       AS nb_customers,
       ROUND(AVG(nb_reservations), 1) AS avg_reservations,
       ROUND(AVG(nb_orders), 1)       AS avg_orders,
```
