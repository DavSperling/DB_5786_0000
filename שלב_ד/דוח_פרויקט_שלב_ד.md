## 🎯 Introduction & objectifs

L'étape **שלב ד** consiste à enrichir la base intégrée des étapes précédentes
(`restaurant_db` — Order & Billing + Customer / Loyalty) avec une couche
**PL/pgSQL** : 2 fonctions, 2 procédures, 2 triggers (dont un sur `UPDATE`), et
2 programmes principaux qui démontrent l'orchestration des objets.

Chaque programme combine plusieurs éléments du langage afin d'obtenir le
maximum de points : curseurs implicites/explicites, **REF CURSOR**, DML
multiples, branchements, boucles, exceptions et records.

| Type        | Fichier                                          | Élément phare                         |
|-------------|---------------------------------------------------|---------------------------------------|
| ALTER       | `AlterTable.sql`                                  | 2 nouvelles tables de support         |
| FUNCTION 1  | `f1_get_top_loyalty_customers.sql`                | RETURN REFCURSOR                      |
| FUNCTION 2  | `f2_calculate_period_revenue.sql`                 | Curseur explicite + LOOP/EXIT         |
| PROCEDURE 1 | `p1_apply_loyalty_tier_discount.sql`              | Multi-DML + CASE                      |
| PROCEDURE 2 | `p2_generate_monthly_waiter_report.sql`           | Curseur implicite (FOR rec IN …)      |
| TRIGGER 1   | `T1_validate_payment_amount.sql`                  | BEFORE INSERT/UPDATE — exception     |

| MAIN 1      | `Main1_LoyaltyRewardWorkflow.sql`                 | Appelle F1 + P1                       |
| MAIN 2      | `Main2_MonthlyRevenueWorkflow.sql`                | Appelle F2 + P2                       |

## 🟦 Fonction F1 — `get_top_loyalty_customers`

### Description

Pour un **tier de fidélité** donné (`Bronze`/`Silver`/`Gold`/`Platinum`)
et un **nombre minimal de commandes**, renvoie un **REF CURSOR** ouvert
sur les meilleurs clients de ce tier, classés par revenu décroissant,
avec une **catégorie de récompense** calculée par `CASE`
(`GOLD_REWARD` / `SILVER_REWARD` / `BRONZE_REWARD` / `NO_REWARD`).

Élément central : retour d'un **REF CURSOR**, consommé par `FETCH ALL` côté
appelant. La fonction valide ses paramètres (lève une exception si tier
invalide ou min_orders négatif) et lève également une exception si **aucun**
client ne correspond.

### Code (extrait, voir `f1_get_top_loyalty_customers.sql`)

```sql
CREATE OR REPLACE FUNCTION get_top_loyalty_customers(
    p_tier        VARCHAR,
    p_min_orders  INT,
    p_cursor      REFCURSOR DEFAULT 'top_loyalty_cur'
) RETURNS REFCURSOR
LANGUAGE plpgsql
AS $$
DECLARE
    v_count INT;
BEGIN
    -- 1. Validation
    IF p_tier NOT IN ('Bronze','Silver','Gold','Platinum') THEN
        RAISE EXCEPTION 'Tier invalide "%".', p_tier USING ERRCODE='22023';
    END IF;
    IF p_min_orders < 0 THEN
        RAISE EXCEPTION 'Le nombre minimal de commandes doit être >= 0 (%).',
            p_min_orders USING ERRCODE='22023';
    END IF;

    -- 2. Vérifier qu'il existe au moins un client matching
    SELECT COUNT(*) INTO v_count
      FROM customer c JOIN loyalty l ON c.customer_id = l.customer_id
                      JOIN loyalty_tier lt ON l.tier_id = lt.tier_id
     WHERE lt.level = p_tier;
    IF v_count = 0 THEN
        RAISE EXCEPTION 'Aucun client trouvé pour le tier "%".', p_tier
            USING ERRCODE='P0002';
    END IF;

    -- 3. Ouvrir le REF CURSOR
    OPEN p_cursor FOR
        SELECT c.customer_id,
               c.first_name||' '||c.last_name        AS customer_name,
               lt.level                              AS tier_level,
               l.points                              AS loyalty_points,
               COUNT(DISTINCT o.order_id)            AS nb_orders,
               COALESCE(SUM(b.final_amount),0)       AS total_revenue,
               CASE
                   WHEN COALESCE(SUM(b.final_amount),0) >= 2000 THEN 'GOLD_REWARD'
                   WHEN COALESCE(SUM(b.final_amount),0) >= 1000 THEN 'SILVER_REWARD'
                   WHEN COALESCE(SUM(b.final_amount),0) >    0 THEN 'BRONZE_REWARD'
                   ELSE                                          'NO_REWARD'
               END                                   AS reward_category
          FROM customer c
          JOIN loyalty l        ON c.customer_id = l.customer_id
          JOIN loyalty_tier lt  ON l.tier_id     = lt.tier_id
          LEFT JOIN "ORDER" o   ON c.customer_id = o.customer_id
          LEFT JOIN bill b      ON o.order_id    = b.order_id
         WHERE lt.level = p_tier
         GROUP BY c.customer_id, c.first_name, c.last_name, lt.level, l.points
        HAVING COUNT(DISTINCT o.order_id) >= p_min_orders
         ORDER BY total_revenue DESC, l.points DESC;

    RETURN p_cursor;
EXCEPTION
    WHEN OTHERS THEN
        RAISE NOTICE 'get_top_loyalty_customers : ERREUR % - %', SQLSTATE, SQLERRM;
        RAISE;
END;
$$;
```

## 🟦 Fonction F2 — `calculate_period_revenue`

### Description

Calcule le **revenu net** (somme des `final_amount`) du restaurant pour une
période `[p_start, p_end]` en parcourant **explicitement** les factures.
Les commandes `Cancelled` sont exclues de l'addition mais comptées.
La fonction lève une exception si aucune facture n'est exploitable
(`P0002`, équivalent `NO_DATA_FOUND`) ou si les bornes sont invalides.

### Code (extrait)

```sql
CREATE OR REPLACE FUNCTION calculate_period_revenue(p_start DATE, p_end DATE)
RETURNS NUMERIC LANGUAGE plpgsql AS $$
DECLARE
    cur_bills CURSOR (s DATE, e DATE) FOR
        SELECT b.bill_id, b.final_amount, o.order_status, o.order_time
          FROM bill b JOIN "ORDER" o ON b.order_id = o.order_id
         WHERE o.order_time::DATE BETWEEN s AND e
         ORDER BY o.order_time;
    rec_bill   RECORD;
    v_total    NUMERIC(14,2) := 0;
    v_kept     INT := 0;
    v_skipped  INT := 0;
    v_avg      NUMERIC(14,2);
BEGIN
    IF p_start IS NULL OR p_end IS NULL THEN
        RAISE EXCEPTION 'Bornes NULL.' USING ERRCODE='22023';
    END IF;
    IF p_end < p_start THEN
        RAISE EXCEPTION 'Période invalide.' USING ERRCODE='22023';
    END IF;

    OPEN cur_bills(p_start, p_end);
    LOOP
        FETCH cur_bills INTO rec_bill;
        EXIT WHEN NOT FOUND;
        IF rec_bill.order_status = 'Cancelled' THEN
            v_skipped := v_skipped + 1;
            CONTINUE;
        END IF;
        v_total := v_total + COALESCE(rec_bill.final_amount, 0);
        v_kept  := v_kept + 1;
    END LOOP;
    CLOSE cur_bills;

    IF v_kept = 0 THEN
        RAISE EXCEPTION 'Aucune facture exploitable entre % et % (skipped=%).',
            p_start, p_end, v_skipped USING ERRCODE='P0002';
    END IF;
    v_avg := v_total / v_kept;
    RAISE NOTICE 'Période %..% : factures gardées=%, ignorées=%, revenu=% (avg=%)',
                 p_start, p_end, v_kept, v_skipped, v_total, v_avg;
    RETURN v_total;
EXCEPTION
    WHEN division_by_zero THEN
        RETURN 0;
    WHEN OTHERS THEN
        RAISE NOTICE 'calculate_period_revenue : ERREUR % - %', SQLSTATE, SQLERRM;
        RAISE;
END;
$$;
```

## 🟧 Procédure P1 — `apply_loyalty_tier_discount`

### Description

Applique une remise supplémentaire aux factures **`In Progress`** des clients
d'un tier de fidélité donné. Le pourcentage demandé est **plafonné** par tier
via un `CASE` :

| Tier     | Plafond |
|----------|---------|
| Bronze   | 10 %    |
| Silver   | 15 %    |
| Gold     | 20 %    |
| Platinum | 30 %    |

Pour chaque facture éligible, la procédure exécute **deux DML** :
1. `UPDATE bill` (mise à jour de `discount_amount` et `final_amount`).
2. `INSERT INTO loyalty_transaction` (1 point fidélité par euro de remise).

Les modifications sont faites dans une boucle qui parcourt **explicitement**
un curseur `FOR UPDATE` (verrouillage de la facture).

### Code (extrait)

```sql
CREATE OR REPLACE PROCEDURE apply_loyalty_tier_discount(
    p_tier      VARCHAR,
    p_extra_pct NUMERIC
) LANGUAGE plpgsql AS $$
DECLARE
    cur_eligible CURSOR (tier_filter VARCHAR) FOR
        SELECT b.bill_id, b.total_amount, b.discount_amount,
               b.final_amount, c.customer_id, l.loyalty_id, lt.level AS tier_level
          FROM bill b
          JOIN "ORDER" o      ON b.order_id    = o.order_id
          JOIN customer c     ON o.customer_id = c.customer_id
          JOIN loyalty l      ON c.customer_id = l.customer_id
          JOIN loyalty_tier lt ON l.tier_id    = lt.tier_id
         WHERE lt.level     = tier_filter
           AND o.order_status = 'In Progress'
         FOR UPDATE OF b;
    rec               RECORD;
    v_max_pct         NUMERIC(5,2);
    v_applied_pct     NUMERIC(5,2);
    v_extra_amount    NUMERIC(10,2);
    v_new_discount    NUMERIC(10,2);
    v_new_final       NUMERIC(10,2);
    v_count           INT := 0;
    v_default_reason  INT;
BEGIN
    IF p_tier NOT IN ('Bronze','Silver','Gold','Platinum') THEN
        RAISE EXCEPTION 'Tier invalide "%".', p_tier USING ERRCODE='22023';
    END IF;
    IF p_extra_pct IS NULL OR p_extra_pct < 0 OR p_extra_pct > 100 THEN
        RAISE EXCEPTION 'Pourcentage invalide.' USING ERRCODE='22023';
    END IF;

    v_max_pct := CASE p_tier
                     WHEN 'Bronze'   THEN 10  WHEN 'Silver'   THEN 15
                     WHEN 'Gold'     THEN 20  WHEN 'Platinum' THEN 30 END;
    v_applied_pct := LEAST(p_extra_pct, v_max_pct);

    SELECT reason_id INTO v_default_reason FROM reason ORDER BY reason_id LIMIT 1;
    IF v_default_reason IS NULL THEN
        RAISE EXCEPTION 'Aucune entrée dans reason.' USING ERRCODE='P0002';
    END IF;

    OPEN cur_eligible(p_tier);
    LOOP
        FETCH cur_eligible INTO rec;
        EXIT WHEN NOT FOUND;
        v_extra_amount := ROUND(rec.total_amount * v_applied_pct / 100, 2);
        v_new_discount := COALESCE(rec.discount_amount,0) + v_extra_amount;
        v_new_final    := GREATEST(rec.total_amount - v_new_discount, 0);

        UPDATE bill SET discount_amount = v_new_discount,
                        final_amount    = v_new_final
         WHERE bill_id = rec.bill_id;

        INSERT INTO loyalty_transaction(transaction_id, points_change, created_at,
                                        loyalty_id, reason_id)
        VALUES ((SELECT COALESCE(MAX(transaction_id),0)+1 FROM loyalty_transaction),
                CEIL(v_extra_amount)::INT, CURRENT_DATE,
                rec.loyalty_id, v_default_reason);
        v_count := v_count + 1;
    END LOOP;
    CLOSE cur_eligible;

    RAISE NOTICE 'apply_loyalty_tier_discount : tier=%, pct_applique=%, factures_mises_a_jour=%',
                 p_tier, v_applied_pct, v_count;
EXCEPTION
    WHEN OTHERS THEN
        RAISE NOTICE 'apply_loyalty_tier_discount : ERREUR % - %', SQLSTATE, SQLERRM;
        RAISE;
END;
$$;
```

## 🟧 Procédure P2 — `generate_monthly_waiter_report`

### Description

Génère (ou régénère) un rapport mensuel par serveur dans la table
`MONTHLY_WAITER_REPORT`. Pour chaque serveur ayant pris des commandes le mois
demandé, on calcule le nombre de commandes, le revenu total, le ticket moyen,
et un niveau de performance (`HIGH ≥ 1500€`, `MEDIUM ≥ 500€`, sinon `LOW`).
Démonstration de la **forme implicite** du curseur : `FOR rec IN <SELECT> LOOP`.

### Code (extrait)

```sql
CREATE OR REPLACE PROCEDURE generate_monthly_waiter_report(p_year INT, p_month INT)
LANGUAGE plpgsql AS $$
DECLARE
    rec               RECORD;
    v_total_revenue   NUMERIC(14,2) := 0;
    v_total_orders    INT           := 0;
    v_perf            VARCHAR(10);
    v_inserted        INT           := 0;
BEGIN
    IF p_year IS NULL OR p_month IS NULL THEN
        RAISE EXCEPTION 'Année et mois NULL.' USING ERRCODE='22023';
    END IF;
    IF p_month < 1 OR p_month > 12 THEN
        RAISE EXCEPTION 'Mois invalide (%).', p_month USING ERRCODE='22023';
    END IF;

    DELETE FROM MONTHLY_WAITER_REPORT
     WHERE report_year = p_year AND report_month = p_month;

    FOR rec IN
        SELECT o.waiter_id,
               COUNT(DISTINCT o.order_id)        AS nb_orders,
               COALESCE(SUM(b.final_amount),0)   AS total_revenue,
               COALESCE(AVG(b.final_amount),0)   AS avg_bill
          FROM "ORDER" o LEFT JOIN bill b ON o.order_id = b.order_id
         WHERE EXTRACT(YEAR  FROM o.order_time) = p_year
           AND EXTRACT(MONTH FROM o.order_time) = p_month
         GROUP BY o.waiter_id ORDER BY total_revenue DESC
    LOOP
        IF    rec.total_revenue >= 1500 THEN v_perf := 'HIGH';
        ELSIF rec.total_revenue >=  500 THEN v_perf := 'MEDIUM';
        ELSE                                  v_perf := 'LOW'; END IF;

        INSERT INTO MONTHLY_WAITER_REPORT(
            report_year, report_month, waiter_id, nb_orders,
            total_revenue, avg_bill, perf_level)
        VALUES (p_year, p_month, rec.waiter_id, rec.nb_orders,
                rec.total_revenue, ROUND(rec.avg_bill,2), v_perf);

        v_total_revenue := v_total_revenue + rec.total_revenue;
        v_total_orders  := v_total_orders  + rec.nb_orders;
        v_inserted      := v_inserted + 1;
    END LOOP;

    IF v_inserted = 0 THEN
        RAISE EXCEPTION 'Aucune commande trouvée pour %-%.', p_year, p_month
            USING ERRCODE='P0002';
    END IF;
    RAISE NOTICE '=== Rapport %-% : % serveurs, % commandes, revenu = % ===',
                 p_year, p_month, v_inserted, v_total_orders, v_total_revenue;
EXCEPTION
    WHEN OTHERS THEN
        RAISE NOTICE 'generate_monthly_waiter_report : ERREUR % - %', SQLSTATE, SQLERRM;
        RAISE;
END;
$$;
```

## 🟥 Trigger T1 — `validate_payment_amount` (BEFORE INSERT/UPDATE)

### Description

`BEFORE INSERT OR UPDATE ON PAYMENT` : empêche d'enregistrer un paiement
dont le `amount` dépasse strictement le `final_amount` de la facture
associée. Si la facture n'existe pas, lève `P0002`. Au passage, complète
`payment_time` avec `CURRENT_TIMESTAMP` quand il est `NULL`.

### Code

```sql
CREATE OR REPLACE FUNCTION fn_validate_payment_amount() RETURNS TRIGGER
LANGUAGE plpgsql AS $$
DECLARE v_bill_total NUMERIC(10,2);
BEGIN
    SELECT b.final_amount INTO v_bill_total FROM bill b WHERE b.bill_id = NEW.bill_id;
    IF NOT FOUND THEN
        RAISE EXCEPTION 'Trigger T1 : la facture % n''existe pas.', NEW.bill_id
            USING ERRCODE='P0002';
    END IF;
    IF NEW.amount > v_bill_total THEN
        RAISE EXCEPTION 'Trigger T1 : paiement % > montant facture % (bill_id=%).',
            NEW.amount, v_bill_total, NEW.bill_id USING ERRCODE='23514';
    END IF;
    IF NEW.payment_time IS NULL THEN
        NEW.payment_time := CURRENT_TIMESTAMP;
    END IF;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_validate_payment_amount
BEFORE INSERT OR UPDATE ON PAYMENT
FOR EACH ROW EXECUTE FUNCTION fn_validate_payment_amount();
```

### Preuve d'exécution


## 🟥 Trigger T2 — `award_loyalty_points_on_bill_update` (AFTER UPDATE)

> Ce trigger satisfait l'exigence du sujet : **au moins un trigger
> en `UPDATE`**.

### Description

`AFTER UPDATE ON BILL` : à chaque modification effective de
`final_amount`, on retrouve le client lié à la facture (via la commande)
et on lui ajoute des points fidélité (1 point par 10 € de la nouvelle
facture). Trois DML enchaînés :

1. `UPDATE loyalty.points` (+ recharge `last_updated`)
2. `INSERT INTO loyalty_transaction` (trace fonctionnelle)
3. `INSERT INTO loyalty_audit_log` (trace technique)

### Code

```sql
CREATE OR REPLACE FUNCTION fn_award_loyalty_points_on_bill_update()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
    rec_link        RECORD;
    v_points        INT;
    v_default_reason INT;
BEGIN
    IF NEW.final_amount IS NOT DISTINCT FROM OLD.final_amount THEN
        RETURN NEW;
    END IF;

    SELECT o.customer_id, l.loyalty_id INTO rec_link
      FROM "ORDER" o LEFT JOIN loyalty l ON l.customer_id = o.customer_id
     WHERE o.order_id = NEW.order_id;

    IF NOT FOUND OR rec_link.customer_id IS NULL OR rec_link.loyalty_id IS NULL THEN
        INSERT INTO loyalty_audit_log(bill_id, customer_id, points_awarded,
                                      old_amount, new_amount)
        VALUES (NEW.bill_id, rec_link.customer_id, 0,
                OLD.final_amount, NEW.final_amount);
        RETURN NEW;
    END IF;

    v_points := GREATEST(FLOOR(NEW.final_amount / 10)::INT, 0);
    IF v_points = 0 THEN
        INSERT INTO loyalty_audit_log(bill_id, customer_id, points_awarded,
                                      old_amount, new_amount)
        VALUES (NEW.bill_id, rec_link.customer_id, 0,
                OLD.final_amount, NEW.final_amount);
        RETURN NEW;
    END IF;

    UPDATE loyalty SET points = points + v_points,
                       last_updated = CURRENT_DATE
     WHERE loyalty_id = rec_link.loyalty_id;

    SELECT reason_id INTO v_default_reason FROM reason ORDER BY reason_id LIMIT 1;
    IF v_default_reason IS NOT NULL THEN
        INSERT INTO loyalty_transaction(transaction_id, points_change, created_at,
                                        loyalty_id, reason_id)
        VALUES ((SELECT COALESCE(MAX(transaction_id),0)+1 FROM loyalty_transaction),
                v_points, CURRENT_DATE, rec_link.loyalty_id, v_default_reason);
    END IF;

    INSERT INTO loyalty_audit_log(bill_id, customer_id, points_awarded,
                                  old_amount, new_amount)
    VALUES (NEW.bill_id, rec_link.customer_id, v_points,
            OLD.final_amount, NEW.final_amount);
    RETURN NEW;
EXCEPTION
    WHEN OTHERS THEN
        RAISE NOTICE 'Trigger T2 : ERREUR % - %', SQLSTATE, SQLERRM;
        RAISE;
END;
$$;

CREATE TRIGGER trg_award_loyalty_points_on_bill_update
AFTER UPDATE ON BILL
FOR EACH ROW EXECUTE FUNCTION fn_award_loyalty_points_on_bill_update();
```

```
---------+--------------+------------

--------------

```

**AFTER :**

```
---------+--------------+-----------
```

## 🟢 Main1 — Workflow récompenses fidélité (F1 + P1)

### Scénario

> Le restaurant veut récompenser ses meilleurs clients **Gold** : on liste
> leur top via la fonction `get_top_loyalty_customers` (F1) puis on
> applique une remise supplémentaire de 5 % à toutes leurs factures en
> cours via `apply_loyalty_tier_discount` (P1). On vérifie l'état BEFORE
> et AFTER, ainsi qu'un appel volontairement erroné pour démontrer le
> bloc EXCEPTION.

### Étapes du programme

1. **BEFORE** : affichage des 10 premières factures `In Progress`
   appartenant à des clients Gold + compteurs `loyalty_transaction` et
   `loyalty_audit_log`.
2. **F1** : `SELECT get_top_loyalty_customers('Gold', 0, 'cur_main1');`
   puis `FETCH ALL IN cur_main1;` (consommation du REF CURSOR).
3. **P1** : `CALL apply_loyalty_tier_discount('Gold', 5);`
4. **AFTER** : mêmes 10 factures (les remises et `final_amount` ont
   changé) + nouveaux compteurs.
5. **EXCEPTION DEMO** : appel de F1 avec un tier invalide `'VIP'`.

### Résultats clés (extraits — texte intégral dans `Main1_LoyaltyRewardWorkflow.sql`)

  14 déclenchements automatiques de T2).
* `loyalty_transaction` passe de **20 000 → 20 027** (+27 lignes : 14 par
  P1 directement + 13 par T2 déclenché en cascade).
* `loyalty_audit_log` passe de **0 → 14** (1 ligne par UPDATE BILL).
* L'exception sur tier `'VIP'` est correctement attrapée :
  `Tier invalide "VIP". Valeurs autorisées : Bronze, Silver, Gold, Platinum.`

✅ Le programme principal démontre l'orchestration complète :
