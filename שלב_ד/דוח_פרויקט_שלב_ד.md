## Introduction & objectives

Stage 4 (**שלב ד**) extends the integrated database from the previous stages
(`restaurant_db` - Order & Billing + Customer / Loyalty) with a **PL/pgSQL**
layer: 2 functions, 2 procedures, 2 triggers (one of them on `UPDATE`), and
2 main programs that orchestrate the objects.

Every program combines several language elements to maximise the score:
implicit and explicit cursors, **REF CURSOR**, multiple DML statements,
branching, loops, exceptions and records.

| Type        | File                                              | Key element                            |
|-------------|---------------------------------------------------|----------------------------------------|
| ALTER       | `AlterTable.sql`                                  | 2 new support tables                   |
| FUNCTION 1  | `f1_get_top_loyalty_customers.sql`                | RETURN REFCURSOR                       |
| FUNCTION 2  | `f2_calculate_period_revenue.sql`                 | Explicit cursor + LOOP/EXIT            |
| PROCEDURE 1 | `p1_apply_loyalty_tier_discount.sql`              | Multi-DML + CASE                       |
| PROCEDURE 2 | `p2_generate_monthly_waiter_report.sql`           | Implicit cursor (FOR rec IN ...)       |
| TRIGGER 1   | `T1_validate_payment_amount.sql`                  | BEFORE INSERT/UPDATE - exception       |

| MAIN 2      | `Main2_MonthlyRevenueWorkflow.sql`                | Calls F2 + P2                          |

## Function F1 - `get_top_loyalty_customers`

### Description

For a given **loyalty tier** (`Bronze`/`Silver`/`Gold`/`Platinum`)
and a **minimum order count**, returns a **REF CURSOR** opened on the top
customers of that tier, ordered by revenue descending, with a derived
**reward category** computed via `CASE`
(`GOLD_REWARD` / `SILVER_REWARD` / `BRONZE_REWARD` / `NO_REWARD`).

The key element is the **REF CURSOR** return value, consumed via
`FETCH ALL` on the caller side. The function validates its arguments
(raises an exception on an invalid tier or a negative `min_orders`) and
also raises an exception when **no** customer matches.

### Code (excerpt, see `f1_get_top_loyalty_customers.sql`)

```sql
CREATE OR REPLACE FUNCTION get_top_loyalty_customers(
    p_tier        VARCHAR,
    p_min_orders  INT,
    p_cursor      REFCURSOR DEFAULT 'top_loyalty_cur'
) RETURNS REFCURSOR
LANGUAGE plpgsql
AS $$
DECLARE
    v_count INT;
BEGIN
    -- 1. Validation
    IF p_tier NOT IN ('Bronze','Silver','Gold','Platinum') THEN
        RAISE EXCEPTION 'Invalid tier "%".', p_tier USING ERRCODE='22023';
    END IF;
    IF p_min_orders < 0 THEN
        RAISE EXCEPTION 'Minimum order count must be >= 0 (got: %).',
            p_min_orders USING ERRCODE='22023';
    END IF;

    -- 2. At least one matching customer?
    SELECT COUNT(*) INTO v_count
      FROM customer c JOIN loyalty l ON c.customer_id = l.customer_id
                      JOIN loyalty_tier lt ON l.tier_id = lt.tier_id
     WHERE lt.level = p_tier;
    IF v_count = 0 THEN
        RAISE EXCEPTION 'No customer found for tier "%".', p_tier
            USING ERRCODE='P0002';
    END IF;

    -- 3. Open the REF CURSOR
    OPEN p_cursor FOR
        SELECT c.customer_id,
               c.first_name||' '||c.last_name        AS customer_name,
               lt.level                              AS tier_level,
               l.points                              AS loyalty_points,
               COUNT(DISTINCT o.order_id)            AS nb_orders,
               COALESCE(SUM(b.final_amount),0)       AS total_revenue,
               CASE
                   WHEN COALESCE(SUM(b.final_amount),0) >= 2000 THEN 'GOLD_REWARD'
                   WHEN COALESCE(SUM(b.final_amount),0) >= 1000 THEN 'SILVER_REWARD'
                   WHEN COALESCE(SUM(b.final_amount),0) >    0 THEN 'BRONZE_REWARD'
                   ELSE                                          'NO_REWARD'
               END                                   AS reward_category
          FROM customer c
          JOIN loyalty l        ON c.customer_id = l.customer_id
          JOIN loyalty_tier lt  ON l.tier_id     = lt.tier_id
          LEFT JOIN "ORDER" o   ON c.customer_id = o.customer_id
          LEFT JOIN bill b      ON o.order_id    = b.order_id
         WHERE lt.level = p_tier
         GROUP BY c.customer_id, c.first_name, c.last_name, lt.level, l.points
        HAVING COUNT(DISTINCT o.order_id) >= p_min_orders
         ORDER BY total_revenue DESC, l.points DESC;

    RETURN p_cursor;
EXCEPTION
    WHEN OTHERS THEN
        RAISE NOTICE 'get_top_loyalty_customers: ERROR % - %', SQLSTATE, SQLERRM;
        RAISE;
END;
$$;
```

## Function F2 - `calculate_period_revenue`

### Description

Computes the **net revenue** (sum of `final_amount`) of the restaurant
for a period `[p_start, p_end]` by walking the bills **explicitly**.
Cancelled orders are excluded from the sum but counted. The function
raises an exception when no exploitable bill exists (`P0002`, equivalent
to `NO_DATA_FOUND`) or when the period bounds are invalid.

### Code (excerpt)

```sql
CREATE OR REPLACE FUNCTION calculate_period_revenue(p_start DATE, p_end DATE)
RETURNS NUMERIC LANGUAGE plpgsql AS $$
DECLARE
    cur_bills CURSOR (s DATE, e DATE) FOR
        SELECT b.bill_id, b.final_amount, o.order_status, o.order_time
          FROM bill b JOIN "ORDER" o ON b.order_id = o.order_id
         WHERE o.order_time::DATE BETWEEN s AND e
         ORDER BY o.order_time;
    rec_bill   RECORD;
    v_total    NUMERIC(14,2) := 0;
    v_kept     INT := 0;
    v_skipped  INT := 0;
    v_avg      NUMERIC(14,2);
BEGIN
    IF p_start IS NULL OR p_end IS NULL THEN
        RAISE EXCEPTION 'Period bounds cannot be NULL.' USING ERRCODE='22023';
    END IF;
    IF p_end < p_start THEN
        RAISE EXCEPTION 'Invalid period.' USING ERRCODE='22023';
    END IF;

    OPEN cur_bills(p_start, p_end);
    LOOP
        FETCH cur_bills INTO rec_bill;
        EXIT WHEN NOT FOUND;
        IF rec_bill.order_status = 'Cancelled' THEN
            v_skipped := v_skipped + 1;
            CONTINUE;
        END IF;
        v_total := v_total + COALESCE(rec_bill.final_amount, 0);
        v_kept  := v_kept + 1;
    END LOOP;
    CLOSE cur_bills;

    IF v_kept = 0 THEN
        RAISE EXCEPTION 'No exploitable bill between % and % (skipped=%).',
            p_start, p_end, v_skipped USING ERRCODE='P0002';
    END IF;
    v_avg := v_total / v_kept;
    RAISE NOTICE 'Period %..%: kept=%, skipped=%, revenue=% (avg=%)',
                 p_start, p_end, v_kept, v_skipped, v_total, v_avg;
    RETURN v_total;
EXCEPTION
    WHEN division_by_zero THEN
        RETURN 0;
    WHEN OTHERS THEN
        RAISE NOTICE 'calculate_period_revenue: ERROR % - %', SQLSTATE, SQLERRM;
        RAISE;
END;
$$;
```

## Procedure P1 - `apply_loyalty_tier_discount`

### Description

Applies an extra discount to **`In Progress`** bills of customers belonging
to a given loyalty tier. The requested percentage is **capped per tier**
through a `CASE`:

| Tier     | Cap |
|----------|-----|
| Bronze   | 10% |
| Silver   | 15% |
| Gold     | 20% |
| Platinum | 30% |

For every eligible bill, the procedure performs **two DML** statements:
1. `UPDATE bill` (updates `discount_amount` and `final_amount`).
2. `INSERT INTO loyalty_transaction` (1 loyalty point per euro of discount).

Updates run inside a loop walking an **explicit** cursor with `FOR UPDATE`
(bill row lock).

### Code (excerpt)

```sql
CREATE OR REPLACE PROCEDURE apply_loyalty_tier_discount(
    p_tier      VARCHAR,
    p_extra_pct NUMERIC
) LANGUAGE plpgsql AS $$
DECLARE
    cur_eligible CURSOR (tier_filter VARCHAR) FOR
        SELECT b.bill_id, b.total_amount, b.discount_amount,
               b.final_amount, c.customer_id, l.loyalty_id, lt.level AS tier_level
          FROM bill b
          JOIN "ORDER" o      ON b.order_id    = o.order_id
          JOIN customer c     ON o.customer_id = c.customer_id
          JOIN loyalty l      ON c.customer_id = l.customer_id
          JOIN loyalty_tier lt ON l.tier_id    = lt.tier_id
         WHERE lt.level     = tier_filter
           AND o.order_status = 'In Progress'
         FOR UPDATE OF b;
    rec               RECORD;
    v_max_pct         NUMERIC(5,2);
    v_applied_pct     NUMERIC(5,2);
    v_extra_amount    NUMERIC(10,2);
    v_new_discount    NUMERIC(10,2);
    v_new_final       NUMERIC(10,2);
    v_count           INT := 0;
    v_default_reason  INT;
BEGIN
    IF p_tier NOT IN ('Bronze','Silver','Gold','Platinum') THEN
        RAISE EXCEPTION 'Invalid tier "%".', p_tier USING ERRCODE='22023';
    END IF;
    IF p_extra_pct IS NULL OR p_extra_pct < 0 OR p_extra_pct > 100 THEN
        RAISE EXCEPTION 'Invalid percentage.' USING ERRCODE='22023';
    END IF;

    v_max_pct := CASE p_tier
                     WHEN 'Bronze'   THEN 10  WHEN 'Silver'   THEN 15
                     WHEN 'Gold'     THEN 20  WHEN 'Platinum' THEN 30 END;
    v_applied_pct := LEAST(p_extra_pct, v_max_pct);

    SELECT reason_id INTO v_default_reason FROM reason ORDER BY reason_id LIMIT 1;
    IF v_default_reason IS NULL THEN
        RAISE EXCEPTION 'No row in reason.' USING ERRCODE='P0002';
    END IF;

    OPEN cur_eligible(p_tier);
    LOOP
        FETCH cur_eligible INTO rec;
        EXIT WHEN NOT FOUND;
        v_extra_amount := ROUND(rec.total_amount * v_applied_pct / 100, 2);
        v_new_discount := COALESCE(rec.discount_amount,0) + v_extra_amount;
        v_new_final    := GREATEST(rec.total_amount - v_new_discount, 0);

        UPDATE bill SET discount_amount = v_new_discount,
                        final_amount    = v_new_final
         WHERE bill_id = rec.bill_id;

        INSERT INTO loyalty_transaction(transaction_id, points_change, created_at,
                                        loyalty_id, reason_id)
        VALUES ((SELECT COALESCE(MAX(transaction_id),0)+1 FROM loyalty_transaction),
                CEIL(v_extra_amount)::INT, CURRENT_DATE,
                rec.loyalty_id, v_default_reason);
        v_count := v_count + 1;
    END LOOP;
    CLOSE cur_eligible;

    RAISE NOTICE 'apply_loyalty_tier_discount: tier=%, applied_pct=%, bills_updated=%',
                 p_tier, v_applied_pct, v_count;
EXCEPTION
    WHEN OTHERS THEN
        RAISE NOTICE 'apply_loyalty_tier_discount: ERROR % - %', SQLSTATE, SQLERRM;
        RAISE;
END;
$$;
```

## Procedure P2 - `generate_monthly_waiter_report`

### Description

Generates (or regenerates) a per-waiter monthly report into
`MONTHLY_WAITER_REPORT`. For every waiter that took orders during the
requested month, we compute the order count, total revenue, average
ticket and a performance level (`HIGH >= 1500 EUR`, `MEDIUM >= 500 EUR`,
otherwise `LOW`). Showcases the **implicit** form of the cursor:
`FOR rec IN <SELECT> LOOP`.

### Code (excerpt)

```sql
CREATE OR REPLACE PROCEDURE generate_monthly_waiter_report(p_year INT, p_month INT)
LANGUAGE plpgsql AS $$
DECLARE
    rec               RECORD;
    v_total_revenue   NUMERIC(14,2) := 0;
    v_total_orders    INT           := 0;
    v_perf            VARCHAR(10);
    v_inserted        INT           := 0;
BEGIN
    IF p_year IS NULL OR p_month IS NULL THEN
        RAISE EXCEPTION 'Year and month NULL.' USING ERRCODE='22023';
    END IF;
    IF p_month < 1 OR p_month > 12 THEN
        RAISE EXCEPTION 'Invalid month (%).', p_month USING ERRCODE='22023';
    END IF;

    DELETE FROM MONTHLY_WAITER_REPORT
     WHERE report_year = p_year AND report_month = p_month;

    FOR rec IN
        SELECT o.waiter_id,
               COUNT(DISTINCT o.order_id)        AS nb_orders,
               COALESCE(SUM(b.final_amount),0)   AS total_revenue,
               COALESCE(AVG(b.final_amount),0)   AS avg_bill
          FROM "ORDER" o LEFT JOIN bill b ON o.order_id = b.order_id
         WHERE EXTRACT(YEAR  FROM o.order_time) = p_year
           AND EXTRACT(MONTH FROM o.order_time) = p_month
         GROUP BY o.waiter_id ORDER BY total_revenue DESC
    LOOP
        IF    rec.total_revenue >= 1500 THEN v_perf := 'HIGH';
        ELSIF rec.total_revenue >=  500 THEN v_perf := 'MEDIUM';
        ELSE                                  v_perf := 'LOW'; END IF;

        INSERT INTO MONTHLY_WAITER_REPORT(
            report_year, report_month, waiter_id, nb_orders,
            total_revenue, avg_bill, perf_level)
        VALUES (p_year, p_month, rec.waiter_id, rec.nb_orders,
                rec.total_revenue, ROUND(rec.avg_bill,2), v_perf);

        v_total_revenue := v_total_revenue + rec.total_revenue;
        v_total_orders  := v_total_orders  + rec.nb_orders;
        v_inserted      := v_inserted + 1;
    END LOOP;

    IF v_inserted = 0 THEN
        RAISE EXCEPTION 'No order found for %-%.', p_year, p_month
            USING ERRCODE='P0002';
    END IF;
    RAISE NOTICE '=== Report %-%: % waiters, % orders, revenue = % ===',
                 p_year, p_month, v_inserted, v_total_orders, v_total_revenue;
EXCEPTION
    WHEN OTHERS THEN
        RAISE NOTICE 'generate_monthly_waiter_report: ERROR % - %', SQLSTATE, SQLERRM;
        RAISE;
END;
$$;
```

## Trigger T1 - `validate_payment_amount` (BEFORE INSERT/UPDATE)

### Description

`BEFORE INSERT OR UPDATE ON PAYMENT`: prevents recording a payment whose
`amount` strictly exceeds the `final_amount` of the linked bill. If the
bill does not exist, raises `P0002`. As a side effect, fills
`payment_time` with `CURRENT_TIMESTAMP` when it is `NULL`.

### Code

```sql
CREATE OR REPLACE FUNCTION fn_validate_payment_amount() RETURNS TRIGGER
LANGUAGE plpgsql AS $$
DECLARE v_bill_total NUMERIC(10,2);
BEGIN
    SELECT b.final_amount INTO v_bill_total FROM bill b WHERE b.bill_id = NEW.bill_id;
    IF NOT FOUND THEN
        RAISE EXCEPTION 'Trigger T1: bill % does not exist.', NEW.bill_id
            USING ERRCODE='P0002';
    END IF;
    IF NEW.amount > v_bill_total THEN
        RAISE EXCEPTION 'Trigger T1: payment % > bill amount % (bill_id=%).',
            NEW.amount, v_bill_total, NEW.bill_id USING ERRCODE='23514';
    END IF;
    IF NEW.payment_time IS NULL THEN
        NEW.payment_time := CURRENT_TIMESTAMP;
    END IF;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_validate_payment_amount
BEFORE INSERT OR UPDATE ON PAYMENT
```

## Trigger T2 - `award_loyalty_points_on_bill_update` (AFTER UPDATE)

> This trigger satisfies the course requirement of having **at least one
> trigger on `UPDATE`**.

### Description

`AFTER UPDATE ON BILL`: every time `final_amount` actually changes, we
locate the customer linked to the bill (through the order) and credit
loyalty points (1 point per 10 EUR slice of the new bill). Three chained
DML statements:

1. `UPDATE loyalty.points` (and refresh `last_updated`)
2. `INSERT INTO loyalty_transaction` (functional trace)
3. `INSERT INTO loyalty_audit_log` (technical trace)

### Code

```sql
CREATE OR REPLACE FUNCTION fn_award_loyalty_points_on_bill_update()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
DECLARE
    rec_link        RECORD;
    v_points        INT;
    v_default_reason INT;
BEGIN
    IF NEW.final_amount IS NOT DISTINCT FROM OLD.final_amount THEN
        RETURN NEW;
    END IF;

    SELECT o.customer_id, l.loyalty_id INTO rec_link
      FROM "ORDER" o LEFT JOIN loyalty l ON l.customer_id = o.customer_id
     WHERE o.order_id = NEW.order_id;

    IF NOT FOUND OR rec_link.customer_id IS NULL OR rec_link.loyalty_id IS NULL THEN
        INSERT INTO loyalty_audit_log(bill_id, customer_id, points_awarded,
                                      old_amount, new_amount)
        VALUES (NEW.bill_id, rec_link.customer_id, 0,
                OLD.final_amount, NEW.final_amount);
        RETURN NEW;
    END IF;

    v_points := GREATEST(FLOOR(NEW.final_amount / 10)::INT, 0);
    IF v_points = 0 THEN
        INSERT INTO loyalty_audit_log(bill_id, customer_id, points_awarded,
                                      old_amount, new_amount)
        VALUES (NEW.bill_id, rec_link.customer_id, 0,
                OLD.final_amount, NEW.final_amount);
        RETURN NEW;
    END IF;

    UPDATE loyalty SET points       = points + v_points,
                       last_updated = CURRENT_DATE
     WHERE loyalty_id = rec_link.loyalty_id;

    SELECT reason_id INTO v_default_reason FROM reason ORDER BY reason_id LIMIT 1;
    IF v_default_reason IS NOT NULL THEN
        INSERT INTO loyalty_transaction(transaction_id, points_change, created_at,
                                        loyalty_id, reason_id)
        VALUES ((SELECT COALESCE(MAX(transaction_id),0)+1 FROM loyalty_transaction),
                v_points, CURRENT_DATE, rec_link.loyalty_id, v_default_reason);
    END IF;

    INSERT INTO loyalty_audit_log(bill_id, customer_id, points_awarded,
                                  old_amount, new_amount)
    VALUES (NEW.bill_id, rec_link.customer_id, v_points,
            OLD.final_amount, NEW.final_amount);
    RETURN NEW;
EXCEPTION
    WHEN OTHERS THEN
        RAISE NOTICE 'T2: ERROR % - %', SQLSTATE, SQLERRM;
        RAISE;
END;
$$;

CREATE TRIGGER trg_award_loyalty_points_on_bill_update
AFTER UPDATE ON BILL
FOR EACH ROW EXECUTE FUNCTION fn_award_loyalty_points_on_bill_update();
```

## Main1 - Loyalty reward workflow (F1 + P1)

### Scenario

> The restaurant wants to reward its top **Gold** customers: list them
> via `get_top_loyalty_customers` (F1) then apply an extra 5% discount
> to all of their `In Progress` bills via `apply_loyalty_tier_discount`
> (P1). The BEFORE/AFTER state is checked, plus a deliberately faulty
> call to demonstrate the EXCEPTION block.

### Steps

1. **BEFORE**: show the 10 first `In Progress` bills owned by Gold
   customers + counters of `loyalty_transaction` and `loyalty_audit_log`.
2. **F1**: `SELECT get_top_loyalty_customers('Gold', 0, 'cur_main1');`
   then `FETCH ALL IN cur_main1;` (REF CURSOR consumption).
3. **P1**: `CALL apply_loyalty_tier_discount('Gold', 5);`
4. **AFTER**: same 10 bills (their discount and `final_amount` have
   changed) + new counters.
5. **EXCEPTION DEMO**: F1 called with an invalid tier `'VIP'`.

### Key results (excerpts - full text in `Main1_LoyaltyRewardWorkflow.sql`)

* F1 returns **128 Gold customers** ordered by revenue descending.
* P1 updates **14 bills** (so 14 INSERTs into `loyalty_transaction` plus
  14 cascaded T2 trigger fires).
* `loyalty_transaction` grows from **20 000 -> 20 027** (+27: 14 from P1
  directly + 13 from T2 fired by the cascade).
* `loyalty_audit_log` grows from **0 -> 14** (one row per `UPDATE BILL`).
* The exception on tier `'VIP'` is correctly caught:
  `Invalid tier "VIP". Allowed values: Bronze, Silver, Gold, Platinum.`

The main program demonstrates the full chain: function -> procedure ->
automatic trigger cascade -> caught exception.

## Main2 - Monthly report workflow (F2 + P2)

### Scenario

> At month-end the manager wants a global report:
> 1. total revenue for the year (F2)
> 2. per-waiter detail, persisted in the database (P2)
> Two extra calls deliberately raise exceptions (empty period for F2,
> empty month for P2) to prove the robustness of the code.

### Steps

1. **DETECTION**: identify the busiest month (`2024-04`, 252 orders) via
   a CTE.
2. **BEFORE**: `MONTHLY_WAITER_REPORT` is empty (`0 rows`).
3. **F2**: `SELECT calculate_period_revenue('2024-01-01','2024-12-31');`
   -> `80 841.18 EUR`.
4. **P2**: `CALL generate_monthly_waiter_report(2024, 4);` -> 228 waiters
   analysed.
5. **AFTER**: top 10 of the report + `perf_level` distribution.
6. **EXCEPTION DEMO 1**: `calculate_period_revenue('1990-01-01','1990-12-31')`.
7. **EXCEPTION DEMO 2**: `generate_monthly_waiter_report(1999, 1)`.

### Key results

* F2 walks 497 bills in 2024 (196 kept + 301 skipped because the orders
  were cancelled) and returns **80 841.18 EUR** of net revenue.
* P2 inserts **228 rows** into `MONTHLY_WAITER_REPORT` for April 2024.
* Distribution per level: 1 HIGH, 92 MEDIUM, 135 LOW.
* Both `EXCEPTION` blocks correctly catch `P0002`:
  > `No exploitable bill between 1990-01-01 and 1990-12-31`
  > `No order found for 1999-1. No report generated.`

Every required language element is showcased: implicit cursor, DML
(DELETE + INSERT), branching IF/ELSIF, exceptions, RAISE NOTICE.
